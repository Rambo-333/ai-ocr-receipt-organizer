# AI-OCR請求書自動処理システム

![Python](https://img.shields.io/badge/python-3.8+-blue.svg)
![PySide6](https://img.shields.io/badge/PySide6-GUI-green.svg)
![EasyOCR](https://img.shields.io/badge/EasyOCR-AI-orange.svg)

月16時間の請求書処理を**98%削減(約15分)**した、AI-OCR自動処理システムです。総務部の手作業による仕分け作業を自動化し、業務効率化を実現しました。

## 📋 概要

**解決した課題:**
- 月間300-400枚の紙請求書を手作業で業者別に仕分け
- 業者ごとにPDFファイル化する作業に**月2日(16時間)**を要していた
- 電子帳簿保存法対応のため、電子化が急務

**実装した解決策:**
- EasyOCR(AI-OCR)で請求書から自動的に情報抽出
- 取引先名や電話番号など記載内容を自動認識
- 業者別にファイル分割し、適切なファイル名で保存
- GUIアプリケーションで直感的な操作を実現

## 💰 ビジネスインパクト

- **工数削減:** 月16時間 → 約15分 (約98%削減)
- **精度向上:** 手作業による仕分けミスを大幅削減
- **法対応:** 電子帳簿保存法への準拠を実現
- **業務効率化:** 請求書仕分けの自動化により月15.75時間を創出

## 📁 プロジェクト構成

```
ReceiptOrganizer/
├── ReceiptOrganizerApp_PySide6.py    # GUIメインアプリケーション
├── ReceiptOrganizerExecutor.py      # OCR処理・PDF分割ロジック
├── config.sample.json                # 設定ファイルテンプレート

├── log_config.json                   # ログ設定

└── README.md
```

**注意:** `poppler/`と`tesseract_bin/`は外部のネイティブバイナリツールです。Pythonパッケージとしてインストールできないため、本リポジトリには含まれていません。セットアップ手順に従って別途ダウンロード・配置してください。この構成により、システムPATHに依存せず、将来的なexe化にも対応可能です。

## 🚀 セットアップ方法

### 前提条件

- Python 3.8以上
- Windows OS (Poppler/Tesseractのパスが前提)

### インストール手順

#### 1. **リポジトリのクローン:**
```bash
git clone https://github.com/Rambo-333/ai-ocr-receipt-organizer.git
cd ai-ocr-receipt-organizer
```

#### 2. **仮想環境の作成:**
```bash
python -m venv venv
venv\Scripts\activate  # Windows
```

#### 3. **依存関係のインストール:**
```bash
pip install -r requirements.txt
```

#### 4. **外部ツールのセットアップ:**

**Poppler (PDF→画像変換):**
- [Poppler for Windows](https://github.com/oschwartz10612/poppler-windows/releases/)をダウンロード
- プロジェクトルートに`poppler/`フォルダを作成し、その中に展開
- `poppler/bin/`ディレクトリが存在することを確認（システムPATHへの追加は不要）

**Tesseract (OCRエンジン):**
- [Tesseract-OCR](https://github.com/UB-Mannheim/tesseract/wiki)をダウンロード
- `tesseract_bin/`フォルダに展開
- 日本語データ(jpn.traineddata)をtessdata/に配置

#### 5. **設定ファイルの作成:**
```bash
copy config.sample.json config.json
copy company_info.sample.CSV company_info.CSV
```

#### 6. **config.jsonの編集:**
```json
{
  "own_company_info": {
    "name": "あなたの会社名",
    "tel": ["0000000000"],
    "fax": ["0000000001"]
  },
  "paths": {
    "poppler_path": "./poppler/bin",
    "tesseract_cmd": "./tesseract_bin/tesseract.exe"
  }
}
```

#### 7. **取引先情報の登録:**
`company_info.CSV`に取引先情報を登録:
```csv
Company,tel,fax,terms
取引先A株式会社,0312345678,0312345679,
取引先B株式会社,0698765432,0698765433,1234567890123
```

#### 8. **アプリケーションの起動:**
```bash
python ReceiptOrganizerApp_PySide6.py
```

## 📊 主要機能

### 1. AI-OCR自動認識
- EasyOCRで請求書全体をスキャン
- 取引先名を自動抽出(正規表現+AI)
- 金額・日付・注文番号も認識(拡張可能)

### 2. 自動ファイル分割
- 一括スキャンしたPDFを自動的にページ分割
- 取引先ごとにファイル名を生成
  - 例: `2025-01-15_サンプル商事株式会社_123456.pdf`
- 出力フォルダに整理して保存

### 3. GUIアプリケーション
- ファイル選択・出力先指定
- リアルタイム進捗表示
- ログウィンドウで処理状況を確認
- エラー時の詳細メッセージ表示

### 4. 取引先データベース管理
- CSV形式で取引先情報を管理
- 電話番号・FAX番号での照合
- 会社名のバリエーション対応

## 🔧 処理フロー

```
1. 請求書PDF一括スキャン
   ↓
2. PDFを1ページずつ画像化(Poppler)
   ↓
3. 各ページをOCRで読み取り(EasyOCR/Tesseract)
   ↓
4. 取引先名・電話番号を抽出
   ↓
5. company_info.CSVと照合
   ↓
6. 取引先別にファイル名生成
   ↓
7. 出力フォルダに保存
```

## 🎯 技術的なポイント

### AI-OCRの実装

EasyOCRで画像からテキストを抽出し、正規表現で取引先名を識別します。日本語の会社形態(株式会社、有限会社等)をパターンマッチングで検出。

詳細な実装はソースコード(`ReceiptOrganizerExecutor.py`)を参照してください。

### マルチスレッド処理
- Pythonの標準`threading`モジュールと`concurrent.futures.ThreadPoolExecutor`で並列処理
- UIをブロックせずバックグラウンドで処理
- PySide6のシグナル経由で進捗をリアルタイム更新

### エラーハンドリング
- OCR失敗時の代替処理
- PDF読み込みエラーのログ記録
- 未登録の取引先への対応

## 💡 今後の拡張可能性

現在の実装は、実務での迅速な課題解決を優先した結果、いくつかの改善点を残しています。これらは、本プロジェクトのさらなる発展可能性を示唆しています。

### 1. ユニットテストの導入による品質向上

現在は手動での動作確認が中心ですが、プロジェクトの保守性と信頼性をさらに高めるため、ユニットテストの導入を計画しています。

*   **対象**: OCR精度に直結する画像前処理ロジックや、多様なフォーマットに対応するための正規表現部分からテストを拡充します。
*   **目的**: 将来的なリファクタリングや機能追加を、既存機能を壊すことなく安全かつ迅速に行える体制を構築します。

### 2. Dockerコンテナ化による環境構築の簡素化

開発過程での試行錯誤の結果、`easyocr`をメインとしつつ`Tesseract`を補助的に用いることで、多様なフォーマットの請求書に対して高い認識精度を達成しました。しかし、これにより`Poppler`と`Tesseract`という外部バイナリへの依存が発生し、環境構築が複雑化しているという課題があります。

*   **解決策**: プロジェクト全体をDockerコンテナ化することで、`pip install`コマンド一発で誰でも同じ環境を再現できるようにします。
*   **効果**: OS依存をなくし、クロスプラットフォーム対応を実現します。

### 3. Pythonライブラリへの依存集約

長期的には、外部ツールへの依存を減らし、`requirements.txt`で管理できるPythonライブラリ（例: `PyMuPDF`など）への移行を検討します。これにより、デプロイメントとメンテナンス性がさらに向上します。これは、ライブラリのOCR精度やパフォーマンスを慎重に評価しながら進めるべき課題です。
